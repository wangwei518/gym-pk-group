# 微信群组运动打卡 PK 小程序产品需求文档（PRS v2）
- 适用平台：微信小程序
- 后端方案：微信云开发（免费版数据库优先，按用户规模评估是否需要优化）
- 运动项目：引体向上 / 跳绳 / 攀岩 / 跑步
- 排行范围：仅展示用户已加入群组内的成员数据，不展示群组外用户信息

## 1. 项目概述
### 1.1 项目背景
为提升用户日常运动积极性，通过群组内好友之间的运动数据排行与 PK 形成正向激励，开发一款轻量化运动打卡微信小程序。用户可每日进行多项目、多次运动打卡，并在主页面查看当前群组内同项目成员累计排名与进度对比。

### 1.2 产品目标
- 支持微信登录，按微信小程序标准流程获取用户身份信息
- 支持引体向上、跳绳、攀岩、跑步四类运动打卡
- 每日可打卡多种项目，同一项目支持多次打卡
- 用户通过分享进入小程序并加入群组，与群组内好友进行 PK
- 主页面可切换项目，展示当前群组内成员累计数值排行
- 数据存储使用微信云开发，优先采用免费资源，在小规模场景下控制成本并保持稳定
- 界面简洁、操作简单、体验流畅、便于后续扩展

### 1.3 用户群体
- 有固定运动习惯的用户
- 希望与朋友、同事、同学在群组内互相监督和比拼运动数据的用户
- 偏好轻量化工具，不想安装独立 App 的用户

### 1.4 MVP 定位
第一阶段聚焦“群组内打卡与排行”，不做群组外公开排行榜，不做复杂社交关系链，不做补打卡与撤销，先验证群组 PK 场景是否成立。

## 2. 页面结构
- 布局（上下布局）
  - 主页面
    - 排行榜
    - 个人页面（我的）
  - 底部 Tab（切换主页面结构）
- 交互
  - 同时只显示排行榜和个人页面其中一个，用底部 Tab 切换
  - 底部 Tab 选中的页面高亮

### 2.1 排行榜页
- 布局（上下布局）
  - 当前群组信息区
    - 群组名称
    - 成员数量
    - 邀请好友加入入口
  - 次级 Tab，切换运动项目
    - 引体向上
    - 跳绳
    - 攀岩
    - 跑步
  - 排行榜列表（显示当前群组内条目，按当前项目累计总数降序排列）
    - 条目（用户头像 + 用户昵称 + 进度条 + 右侧显示该项目累计总数及单位）
  - 底部固定按钮（手动打卡入口，默认带入当前选中项目）

### 2.2 个人页面（我的）
- 布局
  - 用户头像
  - 昵称
  - 当前所在群组信息
  - 四项运动累计数据概览
  - 历史打卡记录列表（按时间倒序）

### 2.3 加入群组流程页
- 展示分享进入后的群组基础信息
- 用户登录后可确认加入群组
- 已在群组内则直接进入排行榜页
- 未登录时先完成微信登录流程

## 3. 功能需求
### 3.1 用户登录与信息
- 用户首次进入小程序后，按微信小程序标准流程完成登录
- 系统通过登录态获取用户唯一身份标识（如 openid）
- 昵称、头像按微信允许的用户信息获取流程处理，不要求静默获取
- 首次登录自动创建用户信息并保存至云数据库
- 用户信息在同一微信身份下全局唯一，更换设备后不丢失

### 3.2 群组与好友加入
- 用户可创建一个 PK 群组
- 用户可通过分享小程序页面或分享卡片邀请好友加入群组
- 好友通过分享链接进入小程序，完成登录后加入对应群组
- 排行榜仅展示当前群组内成员数据，不展示任何群组外用户信息
- 一个用户可加入一个或多个群组（如第一阶段要降低复杂度，也可限制为仅加入一个群组，需在开发阶段确认）

### 3.3 运动打卡
- 支持项目：
  - 引体向上（单位：次，仅允许正整数）
  - 跳绳（单位：次，仅允许正整数）
  - 攀岩（单位：次，仅允许正整数）
  - 跑步（单位：公里，仅允许大于 0 的数字，最多保留 1 位小数）
- 每日可打卡多个不同项目
- 同一项目一天内允许多次打卡，累计值自动累加
- 每次打卡记录包括：群组、项目类型、本次数量、打卡时间、打卡日期
- 不支持补打卡，只允许记录当前时间发起的打卡
- 不支持撤销与删除打卡记录
- 不支持非法值，包括 0、负数、空值、超出格式要求的值
- 打卡成功后实时同步至云端，并更新当前群组排行榜与个人累计数据

### 3.4 排行榜
- 顶部项目 Tab 切换后，内容实时刷新
- 按当前群组 + 当前项目展示成员累计数据排行
- 进度条长度与当前第一名数值成比例
- 每条记录右侧显示精确总次数或总公里数
- 仅展示当前群组内已加入成员
- 提供快捷打卡入口，支持从当前项目直接提交打卡

### 3.5 个人中心
- 展示本人四项运动累计数据
- 展示当前所在群组信息
- 展示历史打卡记录
- 展示本人在当前群组各项目中的累计值（如需要，可扩展显示名次）

## 4. 数据与后端方案
### 4.1 后端方案
- 数据库：微信云开发云数据库
- 无需自建服务器
- 第一阶段优先使用云开发免费资源
- 在小规模用户场景下，预计可运行在免费额度内；若用户规模增长，需要通过索引、聚合表、查询优化等手段控制成本与性能

### 4.2 数据库集合设计
#### users（用户表）
| 字段名 | 说明 |
|---|---|
| _id | 系统主键 |
| openid | 微信用户唯一标识 |
| nickName | 昵称 |
| avatarUrl | 头像 |
| status | 用户状态，默认正常 |
| defaultGroupId | 默认进入的群组 ID |
| lastLoginTime | 最近登录时间 |
| createTime | 创建时间 |
| updateTime | 更新时间 |

#### groups（群组表）
| 字段名 | 说明 |
|---|---|
| _id | 群组主键 |
| name | 群组名称 |
| ownerOpenid | 群组创建人 |
| inviteCode | 邀请码/分享标识 |
| memberCount | 成员数量 |
| status | 群组状态 |
| createTime | 创建时间 |
| updateTime | 更新时间 |

#### group_members（群组成员表）
| 字段名 | 说明 |
|---|---|
| _id | 主键 |
| groupId | 群组 ID |
| openid | 用户标识 |
| role | 成员角色，owner/member |
| joinTime | 加入时间 |
| status | 成员状态 |

#### records（打卡记录表）
| 字段名 | 说明 |
|---|---|
| _id | 系统主键 |
| groupId | 群组 ID |
| openid | 用户标识 |
| type | 项目类型：pullup / rope / rock / run |
| value | 本次打卡数值 |
| source | 数据来源，第一阶段固定 manual |
| createTime | 打卡时间 |
| date | 日期字符串，如 2026-04-17 |

#### user_stats（用户累计统计表）
| 字段名 | 说明 |
|---|---|
| _id | 主键 |
| openid | 用户标识 |
| totalPullup | 引体向上累计值 |
| totalRope | 跳绳累计值 |
| totalRock | 攀岩累计值 |
| totalRun | 跑步累计公里数 |
| updateTime | 更新时间 |

#### group_user_stats（群组用户累计统计表）
| 字段名 | 说明 |
|---|---|
| _id | 主键 |
| groupId | 群组 ID |
| openid | 用户标识 |
| totalPullup | 当前群组内引体向上累计值 |
| totalRope | 当前群组内跳绳累计值 |
| totalRock | 当前群组内攀岩累计值 |
| totalRun | 当前群组内跑步累计公里数 |
| updateTime | 更新时间 |

### 4.3 数据模型说明
- `records` 用于保存原始打卡明细，支撑历史记录查询
- `user_stats` 用于保存用户全局累计值，提升个人页查询效率
- `group_user_stats` 用于保存群组内用户累计值，提升排行榜查询效率
- 排行榜优先从 `group_user_stats` 查询，避免每次都对 `records` 做全量聚合
- 需要对 `groupId + openid`、`groupId + type`、`openid + createTime` 等查询维度建立索引

## 5. 非功能需求
### 5.1 性能
- 小程序冷启动 ≤ 3 秒
- 项目 Tab 切换响应 ≤ 1 秒
- 排行榜加载完成 ≤ 2 秒
- 打卡提交并同步 ≤ 2 秒

### 5.2 稳定性
- 核心功能（登录、加入群组、打卡、排行）可用率目标 ≥ 99.9%
- 网络异常时提供友好提示，不产生重复提交
- 不出现闪退、卡死、明显数据错乱

### 5.3 兼容性
- 微信基础库 ≥ 2.10.0
- 兼容 iOS / Android 主流机型
- 适配常规全面屏、刘海屏、挖孔屏

### 5.4 成本
- 第一阶段优先使用微信云开发免费资源
- 随用户规模增长，需评估数据库读写次数、存储、云函数调用量，并适时优化

## 6. 开发技术栈
- 前端：原生微信小程序 + TypeScript
- UI：官方基础组件 + 自定义排行列表、进度条、群组信息模块
- 存储：微信云开发云数据库
- 逻辑：轻量化封装，结构清晰、易维护

## 7. 开发阶段
1. 环境搭建（1～2 天）
   - 新建小程序项目
   - 开通并初始化云开发环境
   - 创建数据库集合与索引配置

2. 页面结构开发（2 天）
   - 底部 Tab 框架
   - 排行榜页与个人页基础布局
   - 群组加入流程页

3. 核心功能开发（4～6 天）
   - 微信登录与用户信息同步
   - 群组创建、分享、加入逻辑
   - 打卡功能与数据上传
   - 群组内排行榜统计、排序、进度条展示
   - 用户累计数据与历史记录展示

4. 联调与测试（2～3 天）
   - 多用户加入同一群组测试
   - 多次打卡累加逻辑测试
   - 跑步小数输入规则测试
   - 云开发环境稳定性测试
   - BUG 修复与体验优化

## 8. 验收标准
1. 用户可按微信标准流程完成登录并进入小程序
2. 用户可通过分享进入并加入指定群组
3. 四个运动项目可正常切换，数据展示正确
4. 同一项目当天多次打卡后累计值正确
5. 跑步支持 1 位小数，其余三项仅支持正整数
6. 非法输入（0、负数、空值、格式错误）不可提交
7. 排行榜仅展示当前群组成员，不展示群组外数据
8. 个人页面可展示四项累计数据和历史打卡记录
9. 所有数据存储在微信云开发数据库中
10. 更换手机或重装小程序后，重新登录可恢复数据
11. 页面操作流畅，无阻断性缺陷
